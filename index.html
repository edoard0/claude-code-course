Software Architecture Styles

Fundamentals of Software Architecture — Chapters 10–17

Ch. 10 — Layered Architecture

Organises code into horizontal layers; each layer serves the one above it.

graph TD
  A["🖥️ Presentation Layer"]:::blue --> B["⚙️ Business Layer"]:::purple
  B --> C["💾 Persistence Layer"]:::orange
  C --> D["🗄️ Database Layer"]:::teal
  classDef blue fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef purple fill:#d0bfff,stroke:#7c3aed,color:#000
  classDef orange fill:#ffd8a8,stroke:#c2410c,color:#000
  classDef teal fill:#c3fae8,stroke:#0f766e,color:#000

Pros

  • Simple and familiar
  • Very low cost to build
  • Clear separation of concerns
  • No distributed complexity
  • Well-established pattern

Cons

  • Monolithic deployments
  • Hard to scale independently
  • Sinkhole anti-pattern risk
  • Low agility
  • Not fault tolerant

Best Fit

  • Internal business apps
  • Simple CRUD websites
  • Budget-constrained projects
  • Small-team MVPs
  • ERP / back-office tools

Ch. 11 — Pipeline Architecture

Data flows through a series of stateless, single-purpose filters connected by unidirectional pipes.

graph LR
  A["📥 Producer"]:::blue -->|pipe| B["🔄 Transformer"]:::orange
  B -->|pipe| C["✅ Tester"]:::orange
  C -->|pipe| D["➕ Enricher"]:::orange
  D -->|pipe| E["📤 Consumer"]:::green
  classDef blue fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef orange fill:#ffd8a8,stroke:#c2410c,color:#000
  classDef green fill:#b2f2bb,stroke:#15803d,color:#000

Pros

  • Simple and elegant
  • Easy to add new steps
  • Filters are stateless
  • Great for batch processing
  • Low cost

Cons

  • Point-to-point only
  • Not for interactive apps
  • No broadcast capability
  • Limited error handling
  • Sequential by nature

Best Fit

  • ETL data pipelines
  • Unix shell workflows
  • EDI document transforms
  • Log / data processing
  • Apache Camel workflows

Ch. 12 — Microkernel Architecture

A minimal core system extended by independently deployable plug-in components.

graph LR
  P1["🔌 Plugin A"]:::plugin --> CS
  P2["🔌 Plugin B"]:::plugin --> CS
  CS["⚙️ Core System"]:::core --> P3["🔌 Plugin C"]:::plugin
  CS --> P4["🔌 Plugin D"]:::plugin
  CS <--> REG["📋 Plugin Registry"]:::reg
  classDef core fill:#d0bfff,stroke:#7c3aed,color:#000,font-weight:bold
  classDef plugin fill:#ffd8a8,stroke:#c2410c,color:#000
  classDef reg fill:#fff3bf,stroke:#b45309,color:#000

Pros

  • Pluggable and extensible
  • Isolates volatile code
  • Runtime extensibility
  • Easy to customise
  • Low cost for core

Cons

  • Monolithic core
  • Plugin API design is hard
  • Core can become bloated
  • Limited scalability
  • Registry overhead

Best Fit

  • IDEs (VS Code, Eclipse)
  • Tax filing software
  • Insurance claims platforms
  • Product-based SaaS
  • Highly customisable tools

Ch. 13 — Service-Based Architecture

Coarse-grained domain services share a single monolithic database; simpler than microservices.

graph TD
  UI["🖥️ User Interface"]:::blue --> S1["📦 Order Service"]:::svc
  UI --> S2["💳 Payment Service"]:::svc
  UI --> S3["👤 Customer Service"]:::svc
  UI --> S4["📊 Reporting Service"]:::svc
  S1 & S2 & S3 & S4 --> DB[("🗄️ Shared Database")]:::db
  classDef blue fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef svc fill:#d0bfff,stroke:#7c3aed,color:#000
  classDef db fill:#c3fae8,stroke:#0f766e,color:#000

Pros

  • Simpler than microservices
  • Domain isolation
  • Independent deployments
  • DB transactions still possible
  • Good fault tolerance

Cons

  • Shared DB is a bottleneck
  • Coarser granularity
  • Schema change coupling
  • Less elastic
  • Table-sharing risk

Best Fit

  • Financial services apps
  • Healthcare claims
  • Retail order management
  • Mid-size e-commerce
  • CRM systems

Ch. 14 — Event-Driven: Broker Topology

Events are published to a broker; consumers react independently with no central coordinator.

graph LR
  P1["📡 Producer A"]:::prod --> BR{{"📨 Message Broker\n(Topic/Queue)"}}:::broker
  P2["📡 Producer B"]:::prod --> BR
  BR --> C1["⚡ Consumer 1"]:::cons
  BR --> C2["⚡ Consumer 2"]:::cons
  BR --> C3["⚡ Consumer 3"]:::cons
  classDef prod fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef broker fill:#fff3bf,stroke:#b45309,color:#000,font-weight:bold
  classDef cons fill:#b2f2bb,stroke:#15803d,color:#000

Pros

  • Highly decoupled
  • Extremely scalable
  • Async processing
  • Easy to add consumers
  • Fire-and-forget

Cons

  • No workflow visibility
  • Complex error handling
  • Data loss risk
  • Eventual consistency
  • Hard to test / debug

Best Fit

  • IoT event streams
  • Real-time notifications
  • Stock market feeds
  • Audit logging
  • Social media activity

Ch. 14 — Event-Driven: Mediator Topology

A central mediator orchestrates multi-step event workflows with error handling and retries.

graph TD
  IE["📥 Initiating Event"] --> Q["📬 Event Queue"]:::queue
  Q --> M["🎛️ Event Mediator"]:::med
  M --> CH1["📢 Order Channel"]:::ch
  M --> CH2["📢 Payment Channel"]:::ch
  M --> CH3["📢 Email Channel"]:::ch
  CH1 --> EP1["⚙️ Order Processor"]:::proc
  CH2 --> EP2["⚙️ Payment Processor"]:::proc
  CH3 --> EP3["⚙️ Email Processor"]:::proc
  classDef queue fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef med fill:#fff3bf,stroke:#b45309,color:#000,font-weight:bold
  classDef ch fill:#ffd8a8,stroke:#c2410c,color:#000
  classDef proc fill:#b2f2bb,stroke:#15803d,color:#000

Pros

  • Workflow visibility
  • Error handling and retries
  • Transactional control
  • Centralised monitoring
  • Can model complex flows

Cons

  • Mediator = single point of failure
  • More coupled than broker
  • Performance bottleneck
  • Complex to configure
  • God-object risk

Best Fit

  • Order fulfillment workflows
  • Insurance claims processing
  • Healthcare workflows
  • Multi-step booking systems
  • Claims adjudication

Ch. 15 — Space-Based Architecture

Eliminates the DB bottleneck by distributing state across in-memory processing units.

graph TD
  subgraph PUs["Processing Units (elastic)"]
    PU1["🖥️ PU 1\n+ In-Memory Grid"]:::pu
    PU2["🖥️ PU 2\n+ In-Memory Grid"]:::pu
    PU3["🖥️ PU 3\n+ In-Memory Grid"]:::pu
  end
  PUs <--> MW["⚡ Virtualised Middleware\nMessaging Grid | Data Grid | Processing Grid"]:::mw
  MW --> DP["🔄 Data Pump"]:::pump
  DP --> DB[("🗄️ Database")]:::db
  classDef pu fill:#d0bfff,stroke:#7c3aed,color:#000
  classDef mw fill:#fff3bf,stroke:#b45309,color:#000,font-weight:bold
  classDef pump fill:#ffd8a8,stroke:#c2410c,color:#000
  classDef db fill:#c3fae8,stroke:#0f766e,color:#000

Pros

  • Extreme scalability
  • High elasticity
  • No database bottleneck
  • Handles traffic spikes
  • Near-linear scale-out

Cons

  • Very complex to build
  • Expensive infrastructure
  • Data collision risk
  • Hard to test
  • Eventual consistency

Best Fit

  • Concert / event ticketing
  • Online auctions
  • Flash sales
  • Fantasy sports platforms
  • High-volume gaming

Ch. 16 — Orchestration-Driven SOA

Enterprise services reused across the org via a central orchestration engine and service bus.

graph TD
  BS["🏢 Business Services"]:::blue --> OE["🎛️ Orchestration Engine\n(ESB)"]:::orch
  OE --> ES["🔧 Enterprise Services"]:::svc
  OE --> AS["📱 Application Services"]:::svc
  OE --> IS["🛠️ Infrastructure Services"]:::svc
  ES & AS & IS --> DB[("🗄️ Shared DB")]:::db
  classDef blue fill:#a5d8ff,stroke:#2563eb,color:#000
  classDef orch fill:#fff3bf,stroke:#b45309,color:#000,font-weight:bold
  classDef svc fill:#d0bfff,stroke:#7c3aed,color:#000
  classDef db fill:#c3fae8,stroke:#0f766e,color:#000

Pros

  • Enterprise-wide reuse
  • Legacy system integration
  • Centralised governance
  • Unified service contracts
  • Cross-org data sharing

Cons

  • Massive coupling
  • Slow deployments
  • ESB becomes a bottleneck
  • Overly complex
  • Teams blocked on each other

Best Fit

  • Legacy enterprise integration
  • Cross-org data sharing
  • Old ERP integrations
  • ⚠️ Avoid for new systems

Ch. 17 — Microservices Architecture

Fine-grained, independently deployed services each owning their own data store.

graph TD
  GW["🌐 API Gateway"]:::gw
  GW --> MS1["👤 User Svc\n+ DB"]:::svc
  GW --> MS2["📦 Order Svc\n+ DB"]:::svc
  GW --> MS3["💳 Payment Svc\n+ DB"]:::svc
  GW --> MS4["📊 Analytics Svc\n+ DB"]:::svc
  MS1 & MS2 & MS3 & MS4 <--> SM["🕸️ Service Mesh\n(sidecar proxies)"]:::mesh
  classDef gw fill:#a5d8ff,stroke:#2563eb,color:#000,font-weight:bold
  classDef svc fill:#d0bfff,stroke:#7c3aed,color:#000
  classDef mesh fill:#ffd8a8,stroke:#c2410c,color:#000

Pros

  • Extreme scalability
  • Team autonomy
  • Fault isolation
  • Technology diversity
  • Continuous delivery

Cons

  • Distributed system complexity
  • Data consistency is hard
  • High operational overhead
  • Network latency
  • Requires DevOps maturity

Best Fit

  • Netflix / Amazon scale apps
  • Global SaaS platforms
  • Multi-team large orgs
  • High-volume APIs
  • Cloud-native products
